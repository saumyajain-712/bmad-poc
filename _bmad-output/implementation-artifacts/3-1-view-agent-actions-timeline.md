# Story 3.1: View Agent Actions Timeline

Status: review

## Story

As a Developer,  
I want to view a chronological timeline of agent actions for the active run,  
so that I can observe the AI's reasoning process and track progress in real-time.

## Acceptance Criteria

1. **Given** a BMAD run is in progress, **when** I access the run interface, **then** a timeline visually displays each significant agent action (reasoning steps, tool calls) in chronological order.
2. **Given** new run events are generated by backend orchestration, **when** the frontend timeline is open, **then** new events appear within the target latency (use NFR1 <= 1 second as system requirement; treat 500ms mention in epics as stretch target for local demo responsiveness).

## Dev Notes

### Current Implementation Context

- The frontend currently has a single-page shell (`frontend/src/App.tsx`) and main run initiation workflow in `frontend/src/features/run-initiation/RunInitiationForm.tsx`.
- `RunInitiationForm.tsx` already renders `latestRun.context_events` as a plain list. This story should evolve that into a dedicated, chronological timeline component instead of introducing a second event source.
- `frontend/src/services/bmadService.ts` already defines event-like structures under `context_events`; prefer extending these existing interfaces over creating parallel, conflicting models.
- Backend run state and event history are already accessible through `GET /api/v1/runs/{run_id}` in `backend/api/v1/endpoints/runs.py`.

### Technical Requirements

- Follow canonical phase model from `backend/services/orchestration.py`: `("prd", "architecture", "stories", "code")`.
- Respect existing status/event semantics (`phase-status-changed`, `phase-approved`, etc.) from persisted `context_events`; do not invent incompatible event types for core timeline rows.
- Keep API compatibility: if backend changes are required, make them additive and backward-compatible to existing consumers.
- Maintain deterministic ordering behavior for repeated runs (aligned with FR36/FR38 intent).

### Architecture Compliance

- Frontend should stay feature-organized (`frontend/src/features/...`) and keep shared data-access logic in `frontend/src/services/`.
- Use React functional components and TypeScript interfaces; avoid introducing a new state-management library for this story.
- Keep integration boundary clean: UI consumes run/timeline data via service layer; no direct fetches inside deeply nested display components.

### Library / Framework Requirements

- Frontend: React + TypeScript patterns already used in codebase.
- Backend: FastAPI + Pydantic schema contracts must remain source of truth for run payload shape.
- Real-time readiness: design timeline model to support future WebSocket/SSE push events without refactor of rendering component API.

### Testing Requirements

- Verify chronological rendering from seeded event arrays.
- Verify new events appended to timeline without dropping prior entries.
- Verify event rows display phase + event type + timestamp fallback when optional fields are missing.
- Validate no regression in run initiation and clarification flows already covered in backend integration tests.

### Latest Technical Information

- FastAPI WebSocket guidance (current ecosystem references) emphasizes:
  - stable typed event envelopes (`type` + payload),
  - resilient connection management,
  - defensive handling of disconnected clients.
  These patterns should shape timeline event model design even if transport for this story is still polling/snapshot-based.
- React real-time feed guidance emphasizes limiting unnecessary rerenders, append-only updates, and predictable keying for list virtualization readiness.

### Definition of Done

- Event schema consumed by frontend is typed and stable.
- Latency target is measured in local flow and remains within acceptable demo threshold.
- Tests updated/added for core timeline behavior.
- No regressions to existing run initiation and clarification UX.
