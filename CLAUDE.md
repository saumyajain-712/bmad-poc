## Project Overview

**BMAD POC** is a full-stack web app demonstrating AI-driven software development with human-in-the-loop governance. An AI agent executes the BMAD workflow (PRD → Architecture → Stories → Code) while exposing every decision and phase transition for explicit human approval.

## Commands

### Backend
```bash
cd backend

# Run development server (port 8000)
uvicorn main:app --reload

# Run all tests
pytest tests/

# Run a specific test file
pytest tests/test_runs.py -v

# Run a single test
pytest tests/test_runs.py::test_create_run -v
```

### Frontend
```bash
cd frontend

# Run dev server (port 3000, proxies /api/* and /ws/* to backend)
npm run dev

# Run tests
npm run test

# Lint (strict, max-warnings 0)
npm run lint

# Build
npm run build
```

### Full Stack
Start backend first (`uvicorn main:app --reload`), then frontend (`npm run dev`), then open `http://localhost:3000`.

## Architecture

### Data Model
A single `runs` table stores all state. The `Run` model (defined in [backend/sql_app/models.py](backend/sql_app/models.py)) holds:
- `status` — overall run status (`initiated`, `awaiting-clarification`, `in-progress`, `awaiting-approval`, `completed`, etc.)
- `phase_statuses` — per-phase status dict (`pending` → `in-progress` → `awaiting-approval` → `approved`)
- `current_phase` / `current_phase_index` — tracks progress through the phase sequence
- `proposal_artifacts` — stored proposal content for each phase
- `context_events` — append-only event log (the run timeline)
- `resolved_input_context` — merged API specification after clarifications

### Phase Sequence
Phases run strictly in order: `prd` → `architecture` → `stories` → `code`. Each phase must be explicitly approved before the next can start. Phase sequencing logic lives in [backend/services/orchestration.py](backend/services/orchestration.py).

### Request Flow
1. `POST /api/v1/runs/` — validate input completeness; if incomplete, ask clarification questions
2. `POST /api/v1/runs/{id}/clarifications` — merge answers, re-validate
3. `POST /api/v1/runs/{id}/phases/{phase}/start` — generate proposal for the current phase
4. `POST /api/v1/runs/{id}/phases/{phase}/approve` or `/modify` — human governance gate
5. `POST /api/v1/runs/{id}/phases/advance` — move to next phase
6. `POST /api/v1/runs/{id}/resume` — resume from a blocked state after explicit decision

### Adding a Feature
1. Update Pydantic schemas in `backend/sql_app/schemas.py`
2. Add/modify endpoint in `backend/api/v1/endpoints/runs.py`
3. Implement state/persistence logic in `backend/sql_app/crud.py`
4. Add tests in `backend/tests/test_runs.py`
5. Update API client in `frontend/src/services/bmadService.ts`

### Testing Approach
Backend tests use an in-memory SQLite DB (`:memory:`) with `anyio` + `httpx.AsyncClient`. The database dependency is overridden via FastAPI's `override_get_db`. No mocking of CRUD — tests hit a real (in-memory) database. Integration tests in `test_run_integration.py` exercise full phase sequences end-to-end.

### BMAD Artifacts
`_bmad-output/` contains the planning and per-story implementation documents generated by the BMAD methodology. `deferred-work.md` tracks known issues and out-of-scope items. Consult these when context on a story's requirements or prior decisions is needed.
